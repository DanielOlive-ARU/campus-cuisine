# Backend Architecture Sheet

## Project Context
This backend supports a mobile restaurant ordering app built in **.NET MAUI (.NET 10)**. The backend is responsible for:
- serving menu data to the app
- allowing food items to be managed without code changes
- accepting customer orders
- exposing order status
- serving dish images
- validating and sanitising input

The design goal is to minimise custom code by relying on framework features wherever possible.

## Recommended Backend Stack

### Core Stack
- **Python 3.12+**
- **FastAPI** for REST endpoints and OpenAPI/Swagger docs
- **SQLModel** for persistence models and database access
- **SQLite** for lightweight storage
- **Uvicorn** as the ASGI server
- **pydantic-settings** for configuration
- **pytest** + **FastAPI TestClient** for tests

### Why this stack
- FastAPI provides routing, validation, dependency injection, and autogenerated API docs.
- SQLModel keeps the database layer small and readable for coursework.
- SQLite removes infrastructure overhead while satisfying persistent storage requirements.
- Swagger UI at `/docs` can act as the admin CRUD webpage required by the brief.

## Architectural Goals
1. Meet every backend MUST requirement from the brief.
2. Keep the codebase small, readable, and testable.
3. Use framework features instead of custom infrastructure.
4. Keep the API stable for frontend integration.
5. Keep the implementation easy to defend in the viva.

## Backend Responsibilities Mapped to the Brief

| Requirement Area | Backend Response |
|---|---|
| RESTful menu retrieval | `GET /api/menu` and `GET /api/menu/{item_id}` |
| CRUD for food items | Admin endpoints under `/api/admin/menu-items` |
| OpenAPI documented interface/webpage | FastAPI Swagger UI and ReDoc |
| Order status tracking | `POST /api/orders` and `GET /api/orders/{order_id}` |
| Input validation and sanitisation | Pydantic/SQLModel validation + minimal business rules |
| Add food items without code changes | Menu items stored in SQLite and edited through admin CRUD |
| Dynamic dish images | Static image serving with relative image URLs in API responses |
| Testing | pytest suite for health, menu, admin, validation, and orders |

### Folder Roles
- **app/main.py**: app startup, router registration, static files, lifespan hooks
- **app/core/config.py**: environment variables and settings
- **app/core/security.py**: admin API key dependency
- **app/db/engine.py**: SQLite engine and session dependency
- **app/db/init_db.py**: table creation helpers
- **app/db/seed.py**: starter menu data
- **app/models/**: SQLModel entities and shared enums/constants
- **app/schemas/**: request and response models plus validation
- **app/routers/**: route definitions only
- **app/services/**: reusable business logic for menu and orders
- **app/static/images/**: dish image files served to the frontend
- **tests/**: backend automated tests using a separate test database

## Data Model Design

### MenuItem
Represents a food item available to customers.

| Field | Type | Notes |
|---|---|---|
| id | int | primary key |
| name | str | required, trimmed |
| description | str | required |
| category | str/enum | `main`, `dessert`, `appetizer` |
| price | decimal/float | must be greater than 0 |
| image_url | str | relative backend path such as `/images/...` |
| is_available | bool | supports hiding items without deleting |
| created_at | datetime | generated server-side |
| updated_at | datetime | generated server-side |

### Order
Represents a placed customer order.

| Field | Type | Notes |
|---|---|---|
| id | int | primary key |
| status | str/enum | MVP value is `confirmed` |
| created_at | datetime | generated server-side |
| total_items | int | calculated server-side |
| grand_total | decimal/float | calculated server-side |

### OrderLine
Represents one menu item within an order.

| Field | Type | Notes |
|---|---|---|
| id | int | primary key |
| order_id | int | foreign key |
| menu_item_id | int | original menu item id |
| name_snapshot | str | copy of item name at order time |
| unit_price_snapshot | decimal/float | copy of item price at order time |
| quantity | int | must be > 0 |
| line_total | decimal/float | calculated server-side |

### Status Note
The MVP only implements `confirmed` when an order is created. `cancelled` can be added later if a cancel endpoint is introduced.

### Why snapshot fields matter
Order lines should keep a copy of the item name and price at the time the order was placed. That prevents historic orders changing when the menu is edited later.

## API Design

### Health Endpoint
- `GET /health`

Example response:
```json
{
  "status": "ok"
}
```

### Public Endpoints
- `GET /api/menu`
- `GET /api/menu/{item_id}`
- `POST /api/orders`
- `GET /api/orders/{order_id}`

### `GET /api/menu`
Returns available menu items by default.

Optional query parameters:
- `category`
- `available_only`

Example response:
```json
[
  {
    "id": 1,
    "name": "Grilled Chicken Burger",
    "description": "Chargrilled chicken fillet with lettuce and house sauce.",
    "category": "main",
    "price": 8.99,
    "image_url": "/images/grilled-chicken-burger.jpg",
    "is_available": true
  }
]
```

### `GET /api/menu/{item_id}`
Returns one menu item by id.

### `POST /api/orders`
Accepts an order payload from the frontend and creates a new order.

Request shape:
```json
{
  "items": [
    { "menu_item_id": 1, "quantity": 2 },
    { "menu_item_id": 7, "quantity": 1 }
  ]
}
```

Response shape:
```json
{
  "id": 101,
  "status": "confirmed",
  "total_items": 3,
  "grand_total": 21.97,
  "message": "Order placed successfully"
}
```

### `GET /api/orders/{order_id}`
Returns full order details and current status.

Example response:
```json
{
  "id": 101,
  "status": "confirmed",
  "total_items": 3,
  "grand_total": 21.97,
  "created_at": "2026-03-23T12:00:00Z",
  "items": [
    {
      "menu_item_id": 1,
      "name": "Grilled Chicken Burger",
      "unit_price": 8.99,
      "quantity": 2,
      "line_total": 17.98
    },
    {
      "menu_item_id": 7,
      "name": "Chocolate Brownie",
      "unit_price": 3.99,
      "quantity": 1,
      "line_total": 3.99
    }
  ]
}
```

### Admin Endpoints
- `GET /api/admin/menu-items`
- `GET /api/admin/menu-items/{item_id}`
- `POST /api/admin/menu-items`
- `PUT /api/admin/menu-items/{item_id}`
- `DELETE /api/admin/menu-items/{item_id}`

These endpoints are intended for admin CRUD through Swagger UI.

## Validation and Sanitisation Strategy

Use framework validation first, then add minimal custom business rules.

### Validation rules
- `name` must not be blank
- `description` must not be blank
- `category` must be one of the allowed values
- `price` must be greater than 0
- `quantity` must be greater than 0
- orders must contain at least one line item
- referenced menu item ids must exist
- unavailable menu items cannot be ordered

### Sanitisation rules
- trim whitespace on text fields
- collapse repeated spaces in names if needed
- set max lengths for names and descriptions
- reject malformed image paths

### Principle
The backend calculates the authoritative total. The frontend may display a local total during ordering, but the backend returns the final truth when the order is placed.

## Authentication / Access Control
The coursework treats role-based access control as a SHOULD, so the MVP should use the smallest sensible approach:
- public menu and order routes are open
- admin routes require the `x-admin-key` header
- the API key is loaded from `.env`

## Static Image Strategy
Use backend-served static files.

### Approach
- store images in `app/static/images/`
- mount static files at `/images`
- store relative image paths in the database, for example `/images/classic-cheesecake.jpg`
- let the frontend combine the backend base URL with the relative path

## Dependency Injection Plan
Use FastAPI dependencies rather than custom plumbing.

### Planned dependencies
- database session dependency
- settings dependency
- admin API key dependency

### Benefits
- less boilerplate
- easy test overrides
- aligns with the coursework requirement for dependency injection

## Business Logic Placement
Keep routers thin.

### Put in routers
- request parsing
- dependency injection
- returning HTTP responses

### Put in services
- menu CRUD behaviour reused by routes
- order total calculation
- order line snapshot creation
- status assignment rules

### Put in schemas/models
- field definitions
- validation rules
- response shape definitions

## Database Choice
### Recommended database: SQLite
Use SQLite because it satisfies persistence requirements without requiring server setup.

## Testing Strategy
### Core backend tests
1. health endpoint returns success
2. menu retrieval returns expected items
3. category filtering works
4. admin routes require the API key
5. admin create/update/delete works
6. invalid menu payloads are rejected
7. valid orders create correct totals
8. invalid orders are rejected
9. order lookup works
10. historic orders remain stable if menu data changes later

### Recommended tools
- `pytest`
- `TestClient`
- a separate SQLite database for tests

## Integration Contract with Frontend
- frontend requests menu data from the backend
- frontend holds the current order locally until checkout
- backend recalculates final totals when the order is placed
- backend returns the authoritative order id, totals, and status
- `image_url` values are relative paths, not full absolute URLs

## Error Handling Approach
Keep error responses simple and consistent.
- `400` for invalid business requests where needed
- `401` for missing or invalid admin API key
- `404` for missing items or orders
- `422` for request-model validation failures handled by FastAPI

## What Not To Build
Avoid these unless all core flows are complete:
- full user authentication
- payment processing
- separate admin frontend
- analytics dashboard
- microservices
- complex repository patterns

## Viva Talking Points for Dan
### Why FastAPI?
It provides REST routing, OpenAPI docs, validation, and dependency injection with little custom code.

### Why SQLite?
It supports editable menu data without adding infrastructure overhead.

### How does this satisfy “add food items without updating code”?
Menu items are stored in SQLite and managed through admin CRUD routes in Swagger UI.

### Why are order totals calculated on the server?
To keep placed orders authoritative and avoid frontend/backend mismatches.
