# Restaurant Flyout Menu App – Project Planning Pack

This repository contains the planning pack and implementation workspace for a 2-person full stack university project using **.NET MAUI** on the frontend and **Python FastAPI** on the backend.

## Team
- **Dan** – Backend lead
- **Adam** – Frontend lead

## Project Summary
The application is a cross-platform restaurant ordering app with flyout navigation. Users browse menu categories, build an order, review the order, and place it through the backend API.

## Current State
- The current frontend implementation lives under `FrontEnd/` and targets **.NET 10**.
- The backend implementation now lives under `backend/`.
- Backend Phase 1 and Phase 2 are implemented and validated locally and in one clean GitHub Actions run.
- Root `docs/` is the source of truth for planning and API documentation.
- Automatic GitHub Actions runs are currently disabled; backend validation is local-first with a manual workflow retained for on-demand CI checks.

## Architecture Summary
- **Frontend:** .NET MAUI, XAML, C#, MVVM, dependency injection
- **Backend:** Python, FastAPI, SQLModel, SQLite, pydantic-settings, pytest
- **Shared Order State:** MAUI singleton/shared service
- **API Contract:** REST + OpenAPI docs generated by FastAPI
- **Images:** dynamically loaded from backend-served relative paths

## Delivery Strategy
1. Implement every MUST requirement end-to-end.
2. Add selected SHOULD features only after the core flow is stable.
3. Attempt MAY features only if the full core system is complete.

## Recommended SHOULD Features
- Dynamic images from the backend
- Consistent branding and reusable styles
- Offline browsing of cached menu data
- Order cancellation before checkout
- Service/data loading tests
- Order state persistence tests
- Scoped GitHub Actions workflows later in the project

## Out of Scope Unless Finished Early
- Full user authentication flow
- Payment gateway integration
- Analytics dashboard
- Complex admin portal UI

## Suggested Milestones
- **Week 1:** planning, wireframes, API contract, repo structure
- **Week 2:** backend scaffold + menu foundations, frontend shell/navigation
- **Week 3:** shared order state + order endpoints + menu pages
- **Week 4:** order summary, place order flow, integration
- **Week 5:** testing, documentation, polish, viva prep

## How To Use This Pack
1. Use the root `docs/` directory as the source of truth.
2. Convert backlog items into GitHub Issues.
3. Implement the backend phase by phase using `docs/api/backend-plan.md`.
4. Use `docs/testing/local-testing-platform.md` for day-to-day backend validation.
5. Keep `docs/api/backend-status.md` updated as milestones are completed.
6. Use `docs/project-management/ci-cd-notes.md` when CI/CD is expanded later.
7. Keep implementation and documentation aligned.
